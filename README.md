# AI 에이전트의 뇌, 신체, 그리고 한계

> 재편본: [llm-field-notes](https://github.com/hojin12312/llm-field-notes) — 이 강의안을 theory · hardware · tools 3개 클러스터로 재구성한 파생 버전입니다.

---

**왜 Claude Code를 배워야 하는가?** 업무 효율 향상은 기본입니다. 코딩 경력이 없어도, 시간이 없어도 — Claude Code를 쓸 줄 알면 머릿속에 있던 것을 직접 만들어낼 수 있습니다. 업무에서도, 개인 프로젝트에서도.

Claude Code는 **코딩에 한해서 압도적인 1황**입니다. 멋진 음악을 작곡하거나, 아름다운 그림을 그리거나, 회로를 설계하지는 못합니다. 그러나 그 일을 해내는 AI를 만들어내는 **최고의 에이전트**입니다. 음악 생성 모델, 이미지 합성 파이프라인, EDA 자동화 도구 — 이 모든 것이 코드 위에 올라가고, 그 코드를 가장 잘 짜주는 것이 Claude Code입니다.

자율적으로 행동하는 AI 에이전트는 어떻게 설계될까요? 그리고 왜 결국 그 한계가 소프트웨어가 아니라 **메모리**에 있을까요? 이 문서는 소프트웨어 아키텍처 관점에서 에이전트를 해부하고, 지능의 진짜 병목이 어디인지 드러냅니다.

## 1. 뇌와 몸: 에이전트의 해부학

ChatGPT에 질문하는 것과 Claude Code가 코드를 짜고 고치는 것은 **체감이 완전히 다릅니다.** 같은 LLM인데 왜 다를까요?

### 순수한 LLM: 손발 없는 뇌

가장 단순한 LLM 사용은 이렇게 생겼습니다:

```
[프롬프트] → ( LLM ) → [텍스트 응답]
```

정교한 두뇌이지만, 완전히 무력합니다.

- 파일을 스스로 열어보지 못합니다.
- 터미널 명령을 실행할 수 없습니다.
- 이전 대화를 기억하지 못합니다 (매번 컨텍스트를 다시 넣어야 함).
- 웹을 검색할 수 없습니다.

> [!NOTE]
> **"그런데 제 ChatGPT는 저를 기억하던데요?"**
> ChatGPT의 기억 기능은 LLM이 스스로 기억하는 게 아닙니다. 대화가 끝날 때 하네스가 중요한 내용을 별도로 저장해 두었다가, 다음 대화를 시작할 때 프롬프트에 몰래 끼워 넣는 방식입니다. LLM 자체는 여전히 그 순간 입력된 컨텍스트만 봅니다.

생각은 할 수 있지만, **아무것도 할 수 없습니다.**

### 에이전트: LLM에 몸을 붙이다

**에이전트(Agent)**는 LLM에 "몸"을 붙인 것입니다. 이 몸에 해당하는 것이 **하네스(harness)** 입니다.

> **하네스(harness)** 란: 원래는 말에 씌우는 마구(馬具). 소프트웨어 맥락에서는 LLM이 외부 세계와 상호작용할 수 있게 감싸주는 실행 환경 — 도구 호출 인터페이스, 반복 루프, 파일 I/O, 안전 장치가 한 덩어리로 묶여 있습니다.

```
                     +---------------------------------+
                     |             Harness             |
                     |                                 |
   [User Input] ---> |   +--------+                    |
                     |   |  LLM   | <-- Tool Result    |
                     |   +----+---+                    |
                     |        | Tool Call              |
                     |        v                        |
                     |   [File / Shell / Web / DB]     |
                     +---------------------------------+
```

하네스가 제공하는 것:
- **도구 호출**: LLM이 "파일 읽어줘", "이 명령 실행해줘"를 구조화된 형식으로 내놓으면, 하네스가 실행 후 결과를 돌려줌
- **반복 루프**: "코드 짜기 → 실행 → 에러 → 수정"을 LLM이 스스로 반복할 수 있게 구동
- **상태 관리**: 세션 동안의 파일 수정, 실행 기록 추적
- **안전 장치**: 위험한 명령 차단, 사용자 승인 요청

### 도구 호출의 실제: 주문서를 쓰는 뇌

이 메커니즘의 핵심을 한 문장으로 압축하면: **LLM은 텍스트만 출력합니다.** 파일을 직접 열거나 명령을 직접 실행하는 코드가 LLM 안에 있는 것이 아닙니다.

**비유:** LLM을 "주문서를 쓸 수 있는 관리자"로 생각하세요. 이 관리자는 직접 창고에 가지 않습니다. "창고 3번 선반 A-4 박스 가져다줘"라고 적힌 메모를 씁니다. 창고 직원(하네스)이 그 메모를 읽고 실제로 가져와서 "A-4 박스 내용물: 나사 50개, 볼트 20개"라고 보고합니다. 관리자는 그 보고를 읽고 다음 지시를 내립니다.

실제 도구 호출 루프는 이렇게 동작합니다:

```
[LLM] "src/main.py를 확인해야겠다"
  → 일반 텍스트 대신 특수 형식 출력:
    { "tool": "Read", "path": "src/main.py" }

[하네스] 이 출력을 가로채 실제 파일 읽기 실행
  → 결과를 컨텍스트에 삽입

[LLM] 파일 내용을 읽고 다음 판단
  → "17번째 줄에 버그. 수정이 필요하다"
    { "tool": "Edit", "file": "src/main.py", ... }

[하네스] 파일 수정 실행 → 완료 보고

[LLM] 보고를 확인
  → "수정 완료. 테스트를 돌려보자"
    { "tool": "Bash", "command": "pytest" }

[하네스] 테스트 실행 → 결과 반환

[LLM] "전부 통과. 완료." → 사용자에게 최종 응답
```

사용자는 최종 응답 한 줄만 봅니다. 그 사이에 수십 번의 도구 호출 루프가 오갔을 수 있습니다. 위 다이어그램에서 "Tool Call" 화살표가 내려가고 "Tool Result"가 올라오는 것이 이 루프입니다.

어떤 도구를 붙이느냐에 따라 에이전트의 능력이 결정됩니다. 파일 읽기·수정, 셸 명령 실행, 웹 검색, DB 조회 — 하네스에 등록된 것이라면 무엇이든 LLM이 주문서를 써서 호출할 수 있습니다.

**LLM은 뇌, 하네스는 몸.** Claude Code, Cursor, Aider, Roo Code — 겉모습이 달라도 개념은 동일합니다.

## 2. 거울 앞의 인공지능: 구조적 치환

AI 에이전트의 구조를 이해하는 가장 빠른 방법은 인간과 대응시켜 보는 것입니다.

### 구조 대응

| 인간 | AI 에이전트 | 차이 |
|---|---|---|
| 뇌 (신경망) | LLM | 인간: 경험으로 실시간 재구성. AI: 학습 완료 후 가중치 동결 |
| 시냅스 (가소성) | 파라미터 | 인간: 평생 변화. AI: 학습 시 결정, 이후 고정 |
| 신체 (실행 인프라) | 하네스 (`zsh`, `python3`, 파일 시스템) | 인간: 유기적으로 진화, 물리 제약 있음. AI: 설계자가 임의로 교체·확장 가능 |

### AI는 인간보다 부족합니다 — 그리고 그래서 도구로 유용합니다

LLM은 인간 뇌와 비교할 때 분명히 열위인 부분이 있습니다. 학습이 끝난 후 가중치가 굳기 때문에 새로운 경험을 흡수하지 못하고, 진짜 이해 없이 패턴을 흉내 내며, 맥락을 벗어나면 쉽게 환각(hallucination)을 일으킵니다.

규모만 봐도 그렇습니다. 현재 가장 큰 프론티어 모델들의 파라미터 수:

| 모델 | 파라미터 수 | 비고 |
|---|---|---|
| Claude Opus 4.7 | ~5조 | Anthropic 비공개, 추정치 |
| GPT-5.4 | 비공개 | OpenAI 비공개, 추정 불가 |
| GLM-5.1 | 7,540억 (MoE) | Zhipu AI, 활성 파라미터 400억. MoE(Mixture of Experts): 입력마다 전체 파라미터 중 일부만 활성화하는 구조 |
| Llama 3.1 | 4,050억 | Meta 공식 발표 |
| **인간 뇌 시냅스** | **약 100조~150조** | 신경과학 측정치 |

단순 규모로는 최대 프론티어 모델도 인간 뇌 시냅스의 수십 분의 1 수준입니다.

**그런데 왜 고수준 작업에서 인상적인 성능을 내는가:** LLM이 학습하는 데이터의 질과 밀도 때문입니다. 인간은 수십 년에 걸쳐 자신이 직접 경험한 것만 배웁니다. LLM은 인류가 기록한 수천억 단어 — 책, 논문, 코드, 웹 문서 — 를 동시에 압축 학습합니다. 시냅스 수는 더 적지만 인류 도서관 전체를 읽은 셈입니다.

**AI 학습 단계를 인간에 빗대면:**

| AI 학습 단계 | 인간의 대응 |
|---|---|
| Pre-training | 유아기~학창 시절: 세상의 모든 텍스트를 읽으며 언어·지식 구조를 내재화 |
| Fine-tuning | 전공 교육·직업 훈련: 특정 도메인에 맞게 행동 방식을 좁혀나감 |
| RLHF | 멘토·사수의 피드백: "이건 좋아, 저건 안 돼"를 반복하며 행동 교정 |
| 학습 완료 후 가중치 | 암묵지(implicit knowledge): 자전거 타는 법처럼 몸이 기억하지만 언어로 설명하기 어려운 것 |

파라미터에 저장된 것은 명시적 사실이 아니라 패턴과 확률 구조입니다. 인간의 암묵지와 성격이 비슷합니다.

**컨텍스트 윈도우는 단기 기억인가:** 맞습니다.

| | 인간 | LLM |
|---|---|---|
| 단기 기억 | 작업 기억(Working Memory), ~7±2 청크 | 컨텍스트 윈도우 (~200K 토큰) |
| 장기 기억 | 수면 중 해마→대뇌피질 공고화 | 파라미터 (학습으로 고정된 가중치) |

결정적 차이: 인간은 단기 기억의 일부가 장기 기억으로 공고화됩니다. LLM은 컨텍스트에서 일어난 일이 **세션이 끝나면 소멸**합니다 — 파라미터에 아무것도 남지 않습니다. 다음 세션의 LLM은 이전 대화를 전혀 모릅니다.

이것이 CLAUDE.md가 존재하는 이유입니다. 세션을 넘겨도 유지해야 할 지식을 파일에 적어두고 매 세션 컨텍스트에 주입하는 방식으로, 외부 메모가 장기 기억을 보조하듯 동작합니다.

### 그래서 왜 유용한가

**그런데 바로 이 "부족함"에서 도구로서의 가치가 생깁니다.**

가중치가 굳어 있다는 것은 동시에 **일관성**을 의미합니다. 인간은 피로, 감정, 경험 편향에 따라 같은 입력에도 다른 판단을 내리지만, LLM은 동일한 입력에 대해 동일한 확률 분포로 응답합니다. 연속해서 변화하는 환경에 적응하는 것보다 일관된 작업에서 일관된 품질의 결과물을 내놓는 것이 중요한 상황에서 이것은 강점입니다.

그리고 몸(하네스)은 인간의 신체와 달리 **설계자가 완전히 통제합니다.** 규칙을 바꾸고 싶으면 CLAUDE.md 한 줄을 수정하면 됩니다. 도구를 추가하고 싶으면 Tool 스펙을 붙이면 됩니다. 인간에게 행동 지침을 바꾸도록 요청하는 것과는 차원이 다른 통제 가능성입니다.

요약하면:

| | 인간 | AI 에이전트 |
|---|---|---|
| 뇌의 유연성 | 높음 (평생 학습) | 낮음 (가중치 고정) |
| 뇌의 일관성 | 낮음 (감정·피로·편향) | 높음 (동일 입력 = 동일 분포) |
| 몸의 확장성 | 물리 제약 있음 | 소프트웨어로 무한 확장 |
| 몸의 통제 가능성 | 낮음 | 설계자가 완전 통제 |

AI 에이전트를 활용하는 핵심 전략은 AI가 인간보다 뛰어난 영역을 찾는 것이 아닙니다. **인간이 직접 하기에 비효율적인 반복·정형 작업을 일관성 있게 수행할 수 있도록, 몸(하네스)을 잘 설계하는 것**입니다.

## 3. 컨텍스트 윈도우 해부

200K 토큰짜리 컨텍스트 윈도우는 균일한 공간이 아닙니다. **레이어가 있습니다.**

```
+====================================================+  <- 200K 토큰 한계 (천장)
|                 Compaction Buffer                  |  압축 버퍼
|    (auto-summarized when context is full)          |  꽉 차면 핵심만 남기고 자동 압축
+----------------------------------------------------+
|                                                    |  남은 공간
|                [ Remaining Space ]                 |
|                                                    |
+----------------------------------------------------+
|                                                    |
|       Conversation History  /  Tool Results        |  대화 기록 / 도구 실행 결과
|    (chat log, file reads, shell output, errors)    |  (동적 — 작업할수록 위로 쌓임)
|                                                    |
+====================================================+  <- 정적 / 동적 경계
|           CLAUDE.md  (global -> project)           |  지침 파일 (전역 -> 프로젝트 순)
+----------------------------------------------------+
|          Skills  (loaded on /skill-name)           |  스킬 파일
+----------------------------------------------------+
|         Tool Specs  (harness definitions)          |  도구 명세 (하네스 정의)
+----------------------------------------------------+
|                   System Prompt                    |  시스템 프롬프트
+====================================================+  <- 컨텍스트 시작 (바닥)
```

### 정적 레이어 — 세션 시작부터 자리를 잡는다

- CLAUDE.md (전역 → 환경 → 프로젝트 지침, 순서대로 누적)
- 호출된 스킬 파일 (`/skill-name` 입력 시 로드)

이 레이어는 세션 내내 메모리에 고정됩니다. **길수록 모든 작업에 메모리를 잡아먹습니다.**

### 동적 레이어 — 작업할수록 위로 쌓인다

- 현재 대화 히스토리
- 도구 실행 결과 (파일 읽기, `zsh` 셸 출력, 에러 로그)
- 코드 실행 결과, 검색 결과

동적 레이어의 핵심 문제는 **정제되지 않은 원본 그대로** 쌓인다는 것입니다. 파일 하나를 열면 파일 전체가, 에러가 나면 스택 트레이스 전체가, 테스트를 돌리면 모든 반복 로그가 그대로 들어옵니다. 대규모 작업에서는 파일 탐색만으로도 컨텍스트를 가득 채울 수 있고, 의미 없는 로그로 가득 찬 컨텍스트는 LLM의 실효 집중력을 떨어뜨립니다.

> 이 문제의 해법이 **서브에이전트(Sub-agent)** 위임입니다. 탐색·반복 실행·에러 테스트처럼 컨텍스트를 많이 소모하는 작업은 서브에이전트에게 맡기고, 메인 에이전트는 깔끔하게 요약된 보고만 받는 구조입니다. 다음 섹션에서 바로 이어집니다.

### Compaction: LLM의 자기 압축

컨텍스트가 한도에 가까워지면 **Compaction** 이 일어납니다.

LLM이 대화 내용을 스스로 요약해 핵심만 남기고 나머지를 버립니다. 정적 레이어는 유지되고, 동적 레이어의 세부 내용이 압축됩니다.

> Compaction 후에는 원래 대화 히스토리가 사라집니다. 오래된 작업을 다시 언급하면 LLM이 기억 못하는 이유입니다.

**실천 원칙**: CLAUDE.md는 짧게 유지하고, 작업은 세션 단위로 쪼개세요. 정적 레이어에 낭비되는 메모리를 최소화하는 것이 곧 지능의 실효 컨텍스트를 늘리는 것입니다.

> **컨텍스트에 못 담는 지식은 어떻게 주입하는가**: 사내 위키·매뉴얼·수백만 개의 로그 — 이 모든 것을 컨텍스트에 통째 넣을 수는 없습니다. 대신 쿼리 시점에 필요한 문서만 검색해서 넣는 구조가 **RAG(Retrieval-Augmented Generation)** 입니다. 컨텍스트 윈도우가 1M으로 열린 2026년에도 RAG가 사라지지 않은 이유는 비용과 정확도 모두 RAG가 유리하기 때문입니다. → [RAG 아키텍처 — 검색·청킹·리랭킹](./appendix/rag-architecture.md)

## 4. 서브에이전트: 컨텍스트를 위임하라

섹션 3에서 확인한 문제 — 동적 레이어는 정제 없이 쌓입니다. 코드베이스를 탐색하면 파일 전체가, 테스트를 돌리면 반복 로그 전체가, 에러가 나면 스택 트레이스 전체가 메인 컨텍스트에 들어옵니다.

대규모 작업이라면 파일 탐색 수백 회, 테스트 수십 번 — 그것만으로 컨텍스트가 가득 찹니다. 관련 없는 정보로 오염된 컨텍스트는 LLM의 실효 집중력을 떨어뜨립니다.

### 해법: 작업 위임, 요약 수신

**서브에이전트(Sub-agent)** 는 메인 에이전트가 생성하는 독립 에이전트입니다. 별도의 컨텍스트 윈도우에서 작업을 수행하고, 완료 후 요약된 결과만 메인에게 반환합니다.

```
[Main Agent]
    |
    | "src/ 전체에서 API 파일 찾아줘"
    v
[Sub-agent]  -- 독립 컨텍스트
  파일 수백 개 탐색
  패턴 매칭 · 로그 처리
    |
    | "routes/api.py, controllers/user.py"
    v
[Main Agent]  -- 컨텍스트에 추가된 것: 요약 두 줄
```

메인 컨텍스트에 들어오는 것은 요약뿐입니다. 서브에이전트가 내부에서 얼마나 많은 파일을 읽고 로그를 처리했든, 메인은 결과만 받습니다.

| | 직접 수행 | 서브에이전트 위임 |
|---|---|---|
| 메인 컨텍스트 증가 | 파일 수백 개 내용 전부 | 요약 단락 하나 |
| 컨텍스트 오염 | 높음 | 없음 |
| LLM 집중력 | 떨어짐 | 유지됨 |

### Claude Code에서의 구현

Claude Code는 `Agent` 도구와 `Task` 도구로 서브에이전트를 지원합니다.

**Agent 도구** — 작업을 위임하고 완료 후 결과를 받는 방식:

```python
Agent({
  description: "코드베이스 탐색",
  prompt: "src/ 디렉토리에서 API 관련 파일을 탐색하고 "
          "파일 경로와 핵심 역할만 요약해 반환해줘. 중간 로그 생략."
})
```

**Task 도구** — 백그라운드에서 비동기로 실행합니다. 시간이 오래 걸리는 테스트·빌드·분석에 적합합니다.

**서브에이전트에 맞는 작업:**
- 코드베이스 탐색 (파일 수백 개 검색)
- 반복 테스트 및 에러 로그 분석
- 독립된 유틸리티 구현
- 보안 검토, 의존성 분석

**핵심**: 서브에이전트 프롬프트에 **출력 형식을 명시**하세요.

> "탐색 후 파일 경로와 핵심 내용만 반환해줘. 중간 과정은 생략."

결과를 통째로 받으면 메인 컨텍스트가 똑같이 오염됩니다. 요약 형식을 프롬프트에서 강제하는 것이 서브에이전트 활용의 핵심입니다.

## 5. 하네스가 체급을 이긴다

"더 큰 모델 = 더 나은 에이전트"는 틀린 공식입니다.

### 모델 크기보다 하네스 품질

| | Big Brain + Weak Body | Sharp Brain + Strong Body |
|---|---|---|
| 모델 크기 예 | GLM-5.1급 (700B+ MoE) | Gemma4:31b급 (30B) |
| 하네스 품질 | 기본 설정, 지침 빈약 | 정교한 지침 주입, 자동화된 오류 감지, 절차 안내 |
| 메모리 부담 | 동일 컨텍스트에서 수배 더 큼 | 가볍고 빠름 |
| 반복 작업 정확도 | 지침을 간혹 무시 | 하네스가 강제, 일관성 높음 |
| 비용 | 높음 | 낮음 |
| **결과** | 느리고 비싸고 불안정 | **빠르고 저렴하고 일관적** |

표에서 보듯, 큰 모델은 KV 캐시가 더 무겁고 비용도 높지만 하네스가 빈약하면 일관성이 떨어집니다. 작은 모델이라도 하네스가 잘 설계되어 있으면 반복 작업에서 더 안정적인 결과를 냅니다.

### 왜 이것이 가능한가

같은 컨텍스트를 채우면 GLM-5.1(700B+ MoE)의 단기 기억은 Gemma4 31B의 수배입니다. 메모리가 더 무겁고, 처리 속도도 느립니다. 반면 Gemma4 31B에 완벽히 설계된 하네스는:

- 반복 지침을 매 호출마다 빠짐없이 주입하고
- 오류를 즉시 감지해 루프를 끊거나 재시도하고
- 복잡한 절차를 단계별로 안내합니다

**모델 업그레이드보다 하네스 정교화가 먼저입니다.**

그리고 하네스의 확장은 여기서 끝나지 않습니다. 외부망 환경에서는 **웹 검색**과 **[MCP 서버](./appendix/claude-code.md)**(Model Context Protocol — 외부 시스템을 Claude의 도구로 연결하는 프로토콜)를 통해 GitHub, Slack, Linear, DB, 브라우저 등 어떤 외부 시스템이든 Claude Code의 네이티브 도구로 편입할 수 있습니다. 코드 작업 중에 이슈를 열고, 결과를 Slack에 보내고, DB를 직접 조회하는 일이 하나의 대화 흐름 안에서 이루어집니다. 폐쇄망 환경에서 웹 검색·MCP가 차단되는 것이 아쉬운 이유가 여기 있습니다.

### 한 단계 더: 에이전트 네트워크를 직접 설계하라

서브에이전트는 Claude Code가 내부에서 처리해주는 방식입니다. 여기서 한 단계 더 나아가면 **Claude Code를 오케스트레이터(orchestrator — 여러 에이전트 노드를 지휘하는 중앙 에이전트)로 삼아 여러 LLM 노드를 직접 설계하고 연결**할 수 있습니다.

각 노드는 독립된 시스템 프롬프트와 하네스를 가집니다. 오케스트레이터가 입력을 중계하고, 조건에 따라 분기하거나 루프를 돌립니다.

```
[Orchestrator (Claude Code)]
          |
          v
[Writer LLM]  -->  [Reviewer LLM]
      ^                  |
      |   (revision)     |   (critique / pass)
      +------------------+
                         |
                   (pass) --> done
```

**작성-검수 루프**가 가장 직관적인 예입니다:

1. Writer LLM이 초안 작성 (시스템 프롬프트: "간결하고 명확하게")
2. Reviewer LLM이 검토 후 피드백 반환 (시스템 프롬프트: "엄격히 비평, 통과 기준 명시")
3. 피드백이 있으면 Writer가 수정 → 2번으로
4. 기준 충족 시 루프 종료

Writer와 Reviewer를 분리하면 **역할 간 충돌이 사라집니다.** 하나의 LLM이 작성과 비평을 동시에 맡으면 자기 결과물에 관대해지는 경향이 있습니다. 역할을 나누면 Reviewer는 오직 비평에만 집중합니다.

모델도 역할에 맞게 배분할 수 있습니다. 비용이 높은 Opus는 검수에만 쓰고 Haiku로 초안을 쓰면 품질을 유지하면서 비용을 줄일 수 있습니다. → [API 비용 최적화 — 모델 라우팅](./appendix/cost-optimization.md#4-모델-라우팅--작은-모델에-먼저-보내라)

이 패턴을 확장하면:

| 구조 | 예시 |
|---|---|
| 작성 → 검수 → 루프 | 문서·코드 생성, 번역 품질 관리 |
| 분석 → 요약 → 검증 | 논문 리뷰, 데이터 분석 보고서 |
| 계획 → 실행 → 반성 | 리팩터링, 시스템 설계 반복 개선 |
| 전문가 패널 (병렬) | 여러 관점 동시 검토 후 합의 |

**구현**: Anthropic Python SDK로 각 노드의 LLM 호출을 스크립트로 작성하고, Claude Code가 이를 오케스트레이션합니다. 또는 Claude Code의 `Agent` 도구를 루프 구조로 연결하는 방법도 있습니다.

각 노드를 맨손으로 짜는 대신 **에이전트 프레임워크**(LangGraph·CrewAI·Claude Agent SDK·OpenAI Agents SDK 등)로 구성할 수도 있습니다. 2026년에는 MCP·A2A·OpenTelemetry 표준이 자리 잡으면서 프레임워크 교체 비용이 크게 낮아졌습니다. → [에이전트 프레임워크 지형도 — 비교와 선택 기준](./appendix/agent-frameworks.md)

각 노드의 시스템 프롬프트 설계가 파이프라인 전체 품질을 결정합니다. 하네스 설계 원칙이 노드 단위로도 그대로 적용됩니다.

그런데 "Reviewer가 제대로 검수하는지"를 어떻게 아나요? 바로 이게 **LLM 평가(eval)** 의 영역입니다. 작성-검수 루프에 오늘 바꾼 프롬프트가 내일 더 나은지, Reviewer judge가 길이·위치·스타일 편향에 흔들리지 않는지 — 숫자로 측정하지 않으면 "좋아진 것 같다"는 느낌에만 기댈 수밖에 없습니다. → [LLM 평가 — 벤치마크·LLM-as-Judge·실무 eval 루프](./appendix/llm-evaluation.md)

## 6. 에이전트 통제술 Part 1: 설정·컨텍스트 관리

> 이 섹션부터 이어지는 원칙들은 주간 70시간 이상 Claude Code를 사용하면서 얻은 경험에서 나왔습니다. 밥 먹을 때, 씻을 때, 자다 일어나서, 회사에서 하루종일, 이동 중에, 화장실에서까지 — 그렇게 몸으로 검증한 패턴들입니다.
>
> 본문에서는 **원칙과 판단 기준**만 짚고, 구체 예시·JSON 조각·세부 절차는 **[부록 §8 — 에이전트 통제 원칙](./appendix/claude-code.md#8-에이전트-통제-원칙)**에 있습니다.

### 원칙 1: 1세션 = 1작업

세션이 길어지면 관련 없는 컨텍스트가 뒤섞이고 Compaction이 중요한 내용을 날립니다. 컨텍스트가 오염됐다 싶으면 `/clear`로 새 세션을 여는 편이 빠릅니다 — 한 번 꼬인 컨텍스트는 수습하려 할수록 더 커집니다.

오염의 주범은 한 세션에서 "이것도 저것도" 섞는 중구난방 요청입니다. Attention이 컨텍스트 전체를 참조하므로, 이는 단순한 집중력 저하를 넘어 연산 비용과 세션 재개 지연으로 이어집니다. → [트랜스포머 추론: Prefill과 Decode](./appendix/transformer-inference.md)

곁가지 질문은 `/btw`, 같은 주제를 두 방향으로 실험하고 싶으면 `/branch`로 분기합니다. 그리고 언제 리셋할지를 직관이 아니라 숫자로 보려면 statusline에 컨텍스트 사용률을 띄워두는 게 좋습니다.

**판단 기준** — 30% 이하 여유 / 60% 이상 마무리 고려 / 80% 이상 새 세션 적극 전환.

> statusline JSON 설정 예시·`/statusline` 자연어 커스터마이즈는 [부록 §8 원칙 1](./appendix/claude-code.md#원칙-1-1세션--1작업).

### 원칙 2: 의존성 경량화

에이전트가 실행 중 읽어들이는 모든 것 — 패키지·파일·도구 출력 — 이 동적 레이어에 쌓입니다. 파일은 필요한 부분만, 셸 출력은 필요한 필드만 `grep`해서 넘기세요. 컨텍스트에 들어오는 정보량을 줄이는 것이 곧 LLM이 집중할 수 있는 유효 공간을 늘리는 것입니다.

### 원칙 3: 메타 최적화에 먼저 투자하라

큰 프로젝트일수록 **설정을 먼저 다듬는 것이 ROI가 압도적으로 높습니다.** 프롬프트에서 매번 설명하고 있는 것이 있다면 CLAUDE.md 규칙 한 줄로 승격시키세요 — 그 한 줄이 수십 번의 반복 지시를 대체합니다. 에이전트와 일하기 전에 "이 에이전트를 어떻게 세팅할 것인가"에 먼저 시간을 쓰는 것이 이후 전체 작업 속도를 결정합니다. → [부록 §3 CLAUDE.md 계층](./appendix/claude-code.md#3-claudemd--계층적-지침-관리)

### 원칙 4: 임시방편(Ad-hoc Patch)을 거부하라

에러가 났을 때 "만약 X 에러가 나면 무시해"를 CLAUDE.md에 추가하고 싶은 유혹이 옵니다. 하지 마세요. 땜빵 규칙이 누적되면 LLM이 모순된 지침 사이에서 충돌을 일으키고, 최악의 경우 환각(hallucination)을 유발합니다. 땜빵은 빚입니다 — 지침은 얇고 모순이 없을수록 좋습니다.

### 원칙 5: CLAUDE.md는 얇게 — 트리거로 지식을 지연 로드하라

CLAUDE.md는 매 세션 통째로 컨텍스트에 올라갑니다. 모든 지식을 직접 쓰지 말고 **트리거**로 위임하세요 — "이 상황이 오면 이 파일을 읽어라"는 조건부 참조만 남기고, 실제 내용은 별도 파일에 둡니다.

```markdown
## 트리거 참조
- API 엔드포인트 작업 시: 반드시 `docs/api-conventions.md`를 먼저 읽어라
- DB 스키마 변경 전: `docs/db-migration-checklist.md` 참조 필수
```

LLM은 이런 자연어 지침을 실제로 준수합니다 — 해당 상황이 오면 Read 도구를 호출해 파일을 열어옵니다. 그 지식은 필요한 세션의 동적 레이어에만 들어오고, 무관한 세션에서는 전혀 로드되지 않습니다. 지침이 얇을수록 LLM이 판단에 집중할 실효 컨텍스트가 늘어납니다.

> 이 패턴을 Claude Code 동작 자체에 메타 적용한 사례(세션 정리·작업 체크리스트 자동화)는 원칙 7과 메타 팁에서 다시 등장합니다.

## 7. 에이전트 통제술 Part 2: 작업 워크플로우

### 원칙 6: 대규모 작업은 체크리스트로 격파하라

연관된 작업이라도 한 세션에 전부 밀어넣으면 컨텍스트가 부족합니다. 먼저 `checklist.md`를 만들고 항목 하나씩 격파하세요. **항목 하나 완료 → 체크 표시 → 커밋 ([커밋이란?](./appendix/git-commit.md)) → 필요하면 새 세션.** 세션마다 항목 하나에 집중하면 각 세션이 가볍게 유지되고 Compaction에 앞선 작업이 사라지는 일도 줄어듭니다.

체크리스트 파일을 CLAUDE.md에 트리거로 연결해두면 다음 세션에서 자동 로드됩니다. → [부록 §8 원칙 6](./appendix/claude-code.md#원칙-6-대규모-작업은-체크리스트로-격파하라)

### 원칙 7: 세션 핸드오프 — CONTINUE.md 패턴

세션을 닫을 때 "문서 정리해줘"라고 하면 Claude Code가 다음 세션용 인수인계 문서를 작성합니다. 다음 세션에서 `@CONTINUE.md` 한 마디면 즉시 브리핑과 다음 할 일을 받을 수 있습니다.

세션을 새로 열지 이어갈지 애매할 때도 직접 물어보면 됩니다 — "지금 세션 정리하고 새로 시작하는 게 나을까, 아니면 여기서 계속하는 게 나을까?" 남은 컨텍스트 여유, 다음 작업과의 연관성, 현재 세션에서 파악한 정보의 재사용 가치를 종합해 권장해줍니다.

**CLAUDE.md vs CONTINUE.md 역할 분리:**

| 파일 | 내용 | 수명 |
|---|---|---|
| `CLAUDE.md` | 영속적 프로젝트 구조·규칙·스타일 | 계속 누적 |
| `CONTINUE.md` | 현재 작업 상태·미해결 이슈·다음 단계 | 매 세션 덮어씀 |

세션 단위 작업 상태를 CLAUDE.md에 쌓지 마세요 — 정적 레이어가 불필요하게 무거워집니다. CONTINUE.md는 매 세션 덮어쓰되, 거기서 배운 영속적 교훈만 CLAUDE.md로 흘려보냅니다.

"문서 정리해줘" 한 마디로 CONTINUE.md가 자동 생성되는 건 전역 CLAUDE.md에 자연어 트리거를 등록해뒀기 때문입니다 — **원칙 5의 트리거 패턴을 Claude Code 동작 자체에 메타 적용한 사례**입니다. → [부록 §8 원칙 7](./appendix/claude-code.md#원칙-7-세션-핸드오프--continuemd-패턴)에 전역 트리거 작성 예시.

### 원칙 8: 계획 모드로 먼저 상의하라

요구사항이 완전히 명확한 경우는 드뭅니다. 바로 실행에 들어가면 엉뚱한 방향으로 구현이 진행돼 수정 비용이 커집니다.

**Plan Mode**는 Claude Code가 파일을 수정하지 않고 구현 계획만 텍스트로 제안하는 모드입니다. `Shift+Tab`으로 토글하거나 `/plan`으로 진입합니다. 이 단계에서 빠뜨린 조건, Claude의 오해, 방향 수정 포인트를 실행 전에 잡을 수 있습니다 — 처음부터 다시 짜는 것보다 계획 단계에서 상의하는 게 훨씬 빠릅니다.

**복잡한 작업일수록, 요구사항이 모호할수록 계획 모드로 시작하세요.** 판단이 서지 않으면 "이 작업, 바로 시작해도 될까 아니면 계획 모드로 먼저 정리하는 게 나을까?" 물어보면 됩니다.

### 원칙 9: 반복·기계적 작업은 스크립트를 먼저 작성하게 하라

같은 패턴 작업을 LLM이 직접 반복하면 파일 하나당 내용 전체가 컨텍스트에 쌓이고, 오류 추적도 어렵습니다. 대신 **"이 작업을 스크립트로 짜줘"** → 검토 → 실행 → 결과만 확인 흐름으로 바꾸세요. LLM이 100개 파일을 직접 처리하면 100번 쌓이지만, 스크립트로 위임하면 결과 한 줄만 들어옵니다.

파일명 일괄 정리·데이터 변환·패턴 치환·로그 분석 같은 정형 작업에 특히 유리합니다. 반대로 파일마다 맥락 판단이 필요한 작업은 서브에이전트 위임(섹션 4)이 낫습니다.

### 원칙 10: 가짜 테스트 입력 생성도 맡겨라

"그럴듯한 가짜 입력"을 손으로 만드는 것은 시간이 걸립니다. Claude Code는 즉시 해줍니다:

> "이 함수 테스트 스크립트 짜줘. 정상 케이스 3개 + 엣지 케이스 2개."  
> "이 JSON 스키마에 맞는 샘플 데이터 100개 생성해줘."

코드 구조를 읽고 "어떤 입력이 의미 있는지" 스스로 판단하기 때문에 가능합니다. 모듈화와 Claude Code를 조합하면 테스트 데이터를 구성하는 부담 없이 테스트 루프를 빠르게 돌릴 수 있습니다.

### 원칙 11: 요청을 쪼개면 결과가 달라진다

"문서 검토해줘"는 너무 넓습니다. 범위가 모호할수록 LLM은 모든 차원을 **얕게** 훑습니다. 목적별로 쪼개면 결과가 달라집니다:

| 모호한 요청 | 구체적인 요청 |
|---|---|
| "문서 검토해줘" | "오탈자·문법만" / "용어 통일만" / "논리 흐름의 전제·모순만" / "섹션 재배치 포인트만" |
| "코드 리뷰해줘" | "보안 취약점만" / "성능 병목만" / "에러 핸들링 누락만" |

좁은 요청은 평가 기준이 명확하므로 그 차원 하나에 집중하는 깊은 패스가 됩니다. 요청을 정확하게 할수록 에이전트는 쓸모있어집니다.

### 메타 팁: 이 모든 작업을 Claude Code에게 맡겨라

CLAUDE.md 슬림화, 체크리스트 작성, 작업 분해 — 이것들을 직접 손으로 할 필요가 없습니다. 현재 상태를 보여주고 의견을 물어보세요.

> "이 CLAUDE.md 너무 길어 보이는데 슬림화할 수 있을까?"  
> "이 작업 목록을 세션 단위로 쪼개서 체크리스트로 만들어줘."

하네스를 설계한 사람의 역할은 **방향을 정하고 의견을 묻는 것**입니다 — 구체적인 파일 편집과 구조화는 Claude Code에게 위임하세요.

그리고 원칙 6~7을 매번 수동으로 부를 필요도 없습니다. 전역 CLAUDE.md에 "구체적 작업 요청이 오면 체크리스트 생성 → 완료 검증 → 세션 전환 판단 제안" 트리거 하나를 등록하면 이 흐름 전체가 자동화됩니다. → [부록 §8 메타 팁](./appendix/claude-code.md#메타-팁-이-모든-작업을-claude-code에게-맡겨라)에 JSON 예시.

## 8. 소프트웨어의 한계, 하드웨어의 가능성

소프트웨어 엔지니어들은 지금 KV 캐시를 쥐어짜며 버티고 있습니다.

- 세션을 쪼갭니다 — KV 캐시가 쌓이지 않게
- CLAUDE.md를 짧게 유지합니다 — 정적 레이어가 무겁지 않게
- Compaction에 의존합니다 — 터지기 전에 스스로 요약하도록

하지만 이 모든 최적화는 **물리적 한계 위에서** 이루어집니다. 아무리 잘 설계된 하네스도 대역폭보다 빠를 수 없고, 아무리 정교한 컨텍스트 관리도 메모리 용량보다 많은 정보를 담을 수 없습니다.

**에이전트의 자아가 깊어지는 것은 컨텍스트 윈도우가 넓어지는 것입니다.** 256K가 1M이 되고 10M이 될 때마다, 소프트웨어 엔지니어들이 상상할 수 있는 에이전트의 반경이 넓어집니다. 그 가능성을 여는 열쇠는 **초대용량, 압도적 대역폭의 메모리**에 있습니다.

지능의 크기는 뇌(파라미터)가 아니라 단기 기억(KV 캐시)을 퍼 나르는 속도가 결정합니다. 그리고 그 속도는 여러분의 손에 달려 있습니다.
