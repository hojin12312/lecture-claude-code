# 커밋(Commit)이란 무엇인가

> 연관 문서: [Claude Code 3대 설정 축](./claude-code.md)

---

## 한 줄 요약

**커밋은 "지금 이 상태를 저장한다"는 스냅샷입니다.**

작업하던 파일들의 변경 내용을 묶어서, 이름(메시지)을 붙여 기록하는 행위입니다. 한 번 커밋된 내용은 나중에 언제든 되돌아볼 수 있고, 이전 상태로 복구할 수도 있습니다.

---

## 비유로 이해하기

게임의 **세이브 포인트**와 같습니다.

- 게임에서 중요한 지점마다 저장해두면, 실수하거나 다른 시도를 해보고 싶을 때 그 지점으로 돌아올 수 있습니다.
- 커밋도 마찬가지입니다. 코드(또는 문서) 작업 중 "여기까지는 확실히 동작한다"는 시점에 커밋해두면, 이후에 무언가 잘못되더라도 이 지점으로 되돌릴 수 있습니다.

---

## Git과 커밋의 관계

커밋은 **Git**이라는 버전 관리 시스템의 핵심 개념입니다. Git은 파일의 변경 이력을 커밋 단위로 쌓아두는 도구입니다.

```
commit 1  "first draft"       <- 첫 초안
  |
commit 2  "add chapter 2"     <- 2장 추가
  |
commit 3  "fix typo"  <-- now <- 오탈자 수정, 현재 위치
```

이 커밋들이 쌓인 공간을 **저장소(repository, 줄여서 repo)**라고 합니다.

---

## 로컬 커밋 vs 푸시(Push)

커밋은 기본적으로 **내 컴퓨터 안**에만 저장됩니다.

다른 사람과 공유하거나 GitHub 같은 원격 저장소에 올리려면 **푸시(push)** 라는 별도 동작이 필요합니다.

```
  My PC (local)                GitHub (remote)
+------------------+          +------------------+
| commit 1         |          | commit 1         |
| commit 2         |  push --> | commit 2         |
| commit 3 (new)   |          |                  |
+------------------+          +------------------+
```

로컬 커밋은 언제든 되돌리거나 수정할 수 있어서 부담이 없습니다. 푸시는 공유 공간에 올라가는 것이라 신중하게 합니다.

---

## Claude Code에서의 커밋

Claude Code는 작업이 완료된 시점에 자동으로 커밋을 만들 수 있습니다. "항목 하나 완료 → 커밋"이라는 워크플로우는 다음을 의미합니다:

- 작업 단위가 명확하게 기록으로 남습니다.
- 무언가 잘못됐을 때 직전 커밋으로 즉시 되돌릴 수 있습니다.
- 세션이 바뀌어도 어디까지 했는지 커밋 이력으로 확인할 수 있습니다.

세이브 없이 긴 게임을 하다 죽으면 처음부터인 것처럼, 커밋 없이 긴 작업을 하다 실수하면 되돌릴 지점이 없습니다.

---

## 되돌리기: 비개발자가 알아야 할 세 가지

커밋이 무서운 이유 중 하나는 "실수하면 되돌릴 수 없을 것 같다"는 느낌 때문입니다. 실제로는 대부분의 실수가 몇 분 안에 복구 가능합니다. 일상 작업의 90%는 아래 세 명령으로 해결됩니다.

### 상황 1. 파일을 저장만 하고 커밋은 아직 안 했는데, 변경을 되돌리고 싶다

```
git restore <파일경로>
```

해당 파일을 **마지막 커밋 상태**로 되돌립니다. 아직 커밋 안 한 내 작업만 사라지고, 기록된 커밋은 건드리지 않습니다.

### 상황 2. 방금 만든 커밋을 취소하고 싶다 (메시지가 틀렸거나 파일을 빠뜨렸을 때)

```
git reset --soft HEAD~1
```

가장 최근 커밋만 취소합니다. **파일 내용은 그대로 유지**되고, "아직 커밋 안 한 변경"으로 다시 돌아갑니다. 수정해서 다시 `git add` → `git commit` 하면 됩니다.

> `--soft`는 "커밋 기록만 지운다"는 뜻입니다. `--hard`는 **파일 내용까지 날리는** 훨씬 위험한 옵션이니 실수로 치지 마세요.

### 상황 3. 이전 커밋 시점의 코드가 잠깐 보고 싶다 (되돌리지는 않고 참고만)

```
git checkout <commit 해시> -- <파일경로>
```

그 파일만 예전 상태로 복원합니다. 다른 파일은 그대로. 참고 후 최신으로 돌아오려면:

```
git checkout HEAD -- <파일경로>
```

## 체크포인트로서의 커밋 — Claude Code 워크플로우

Claude Code는 작업 단위마다 커밋을 만드는 습관에 특히 잘 맞습니다.

### 체크리스트 단위 커밋

긴 작업을 Claude가 여러 단계로 쪼갠 뒤, 각 단계 완료 시마다 커밋:

```
step 1  read auth module         [done]   -->  commit: "read: auth module"
step 2  add input validation     [done]   -->  commit: "feat: add input validation"
step 3  update tests             [done]   -->  commit: "test: add validation tests"
```

각 커밋이 "여기까지는 확실히 동작" 체크포인트입니다. 3번 단계에서 잘못이 드러나면 2번까지만 살리고 다시 시도할 수 있습니다.

### CONTINUE.md와 커밋 히스토리

세션 경계에서 `CONTINUE.md` (다음 세션 지침)를 남기고 커밋하면:

- 새 세션 시작 시 `@CONTINUE.md` 한 마디로 맥락 복구
- 커밋 이력에서 "이번 세션에 어디까지 했는지" 역추적 가능

세션당 CONTINUE.md 1회 갱신 + 마지막 커밋이 기본 패턴입니다.

### Slash Command로 커밋 자동화

`.claude/commands/` 디렉토리에 `commit.md` 같은 파일을 두면 `/commit` 한 번으로 커밋 메시지 작성부터 실행까지 맡길 수 있습니다. "매번 메시지 어떻게 쓰지?" 고민을 줄여줍니다. (자세한 설정 방식은 [Claude Code 레퍼런스](./claude-code.md) 참조)

## 이 레포의 커밋·푸시 정책 (참고 예시)

프로젝트마다 커밋 정책이 다를 수 있다는 걸 보여주는 예시로, 이 강의 자료 레포(`lecture-claude-code`)의 정책을 소개합니다.

- **로컬 커밋 자유** — 언제든 쌓아도 무방. 되돌릴 수 있어 부담 없음.
- **push는 명시 요청 시에만** — 공유 상태는 신중히.
- **README 수정이 포함된 커밋은 즉시 push** — 호진님이 GitHub에서 바로 확인해야 하므로 예외.
- **Orphan push 방식** — GitHub에는 커밋 히스토리를 노출하지 않음. push할 때마다 히스토리를 초기화한 단일 커밋 상태로 업로드.

Orphan push는 일반적이지 않은 정책입니다. 일반 프로젝트는 표준 `git push`로 커밋 이력 전체를 공유하고, 이력 자체가 협업과 디버깅의 자산입니다. 강의 자료 특성상 "과정"보다 "현재 상태"만 노출하려는 의도로 이 레포에서만 채택했습니다.

---

## 정리

- 커밋 = "이 상태 저장" 세이브 포인트.
- **로컬**에만 쌓는 단계: 언제든 되돌릴 수 있어 부담 없음.
- **push**해서 원격에 올라간 시점: 신중해야 함.
- Claude Code 워크플로우에서 **체크리스트 단위 커밋**은 실수 복구의 핵심 장치.
- 비개발자용 3대 복구 명령: `git restore`, `git reset --soft HEAD~1`, `git checkout <hash> -- <file>`.
